Small Model Evaluation

Can free, locally-run language models successfully query these databases? We handed four open-weight models — from 3 B to 36 B parameters — the same instructions any assistant gets, asked twenty research questions spanning both databases, and executed whatever SQL they wrote against the live databases through the public endpoint. No cloud AI involved.

01 Method

Blind questions, real execution, pre-registered answers

Each model ran locally through Ollama at temperature 0 and received a condensed version of the project's skill instructions — schema, exact value vocabularies, corpus rules, and the search-function signatures. It was then asked twenty research questions, one at a time: eleven against the SWRD journal database, nine against the SSWR conference database, covering counts, joins, vocabulary filters, subqueries, and the built-in search functions. The model's SQL was executed verbatim against the live databases through the same public read-only endpoint available to everyone, and compared against a reference answer computed before the run.

Scoring: a strict pass (✓) requires reproducing the reference value exactly. Failures were then inspected individually: answers that differed only because the model applied the documented corpus-scoping rules more strictly than the reference query are marked defensible (△); genuine mistakes are marked ✗.

03 Demonstration

Real transcripts: question in, answer out

Qwen3.6-35B · Q16 · 22.9 s

"Which single institution had the most first-authored SSWR presentations between 2020 and 2026?"

SELECT institution_normalized, COUNT(*) AS count
FROM sswr.papers
JOIN sswr.paper_authors
  ON sswr.papers.id = sswr.paper_authors.paper_id
WHERE year BETWEEN 2020 AND 2026
  AND author_order = 1
GROUP BY institution_normalized
ORDER BY count DESC LIMIT 1

→ Washington University in St. Louis · 290  ✓

Qwen3.6-35B · Q20 · 5.2 s

"What is the earliest SSWR conference year with more than 1,000 presentations?"

SELECT year FROM sswr.papers
GROUP BY year
HAVING COUNT(*) > 1000
ORDER BY year ASC LIMIT 1

→ 2015  ✓

Granite 3B · Q5 · 0.4 s

"How many presentations were given at the 2024 SSWR conference?"

SELECT COUNT(*) FROM sswr.papers WHERE year = 2024

→ 1,516  ✓  (in under half a second, from a 2.1 GB model)

What △ "defensible" looks like — Gemma4 · Q3

"How many SWRD articles from 1989 onward mention 'kinship care'…?"

SELECT count(*) FROM swrd.papers
WHERE publication_year >= 1989
  AND is_scientific = true      -- added by the model
  AND abstract IS NOT NULL      -- added by the model
  AND (title ILIKE '%kinship care%'
       OR abstract ILIKE '%kinship care%')

The model added the skill file's default corpus-scoping filters (224 vs the reference's 239) — a stricter reading, not a mistake.

04 Failure analysis

Every miss, examined

△ Over-applied corpus rules — 3 cells

Models added the documented default filters (1989+, scientific, has-abstract) even when a question said "all years" or asked for a raw substring count. Stricter than asked; arguably better behavior for real research use.

✗ Errors by the 3B — 4 cells

Granite emitted a bare function call without SELECT, twice referenced the year column without joining the table it lives in, and once applied corpus scoping despite the question explicitly asking for the full total — the classic small-model mix of syntax slips and instruction drift.

✗ Logic error — 1 cell (the 20B)

gpt-oss sorted search results by rank ascending, returning the worst match as "top." A subtle, dangerous class of error — syntactically valid, silently wrong.

The pattern across sizes: reliability scales with size (3B: three construction errors; 36B: zero errors of any kind), but even the smallest model was productive — and the mid-size models' only "failures" were excess caution. The precision of the skill files (exact vocabularies, explicit rules, measured coverage) appears to be what makes small models viable here.

05 Honest limits & reproduction

What this shows — and how to rerun it

It shows that free, local models — down to 2 GB — can translate research questions into correct queries against these databases when given the skill files, and that a consumer-hardware 36B model does so flawlessly.

It doesn't show multi-step analysis, semantic-search orchestration, or robustness across phrasings — twenty single-turn questions, one run each, temperature 0. An existence proof with a clear trend, not a leaderboard.

Reproduce it with any Ollama model: python migration/08_eval_local_models.py <model> … — harness, questions, reference queries, and raw outputs (including every model's SQL) are in the repository.
